# Reqtest Workflow – Blekingetrafiken Travel Planner

## 1. Purpose

This document describes how Reqtest is used during the practical test project for Blekingetrafiken's web-based travel planner.

Reqtest is used as the main project tool for structuring test suites, writing test cases, tracking work on the Agile Board, executing tests and reporting bugs.

## 2. Project tool

The project group uses Reqtest for:

- Test suites
- Test cases
- Test execution
- Bug reporting
- Agile Board workflow
- Test progress overview

## 3. Test suite structure

The testing is divided into several test suites based on functional areas in the travel planner.

Current test suite areas include:

| Test suite area | Purpose |
|---|---|
| Search fields | Tests related to entering, selecting and validating locations |
| Result page | Tests related to displayed travel results |
| Technical details | Tests related to technical information and journey details |
| Departure/arrival and return trip | Tests related to time, date, departure, arrival and return journey |
| Language selection | Tests related to changing or using language options |
| Other buttons and links | Tests related to help, print, map and navigation functions |
| Change travel preferences | Tests related to adjusting travel preferences or options |

## 4. Test case structure

Each test case should be clear enough to be executed by another person in the group.

A test case should include:

- Title
- Short description
- Priority
- Preconditions
- Test steps
- Expected result
- Actual result during execution
- Status

## 5. Bug report structure

Bug reports should be written in a way that makes the issue easy to understand, reproduce and prioritize.

A bug report should include:

- Clear title
- Description of the problem
- Steps to reproduce
- Expected result
- Actual result
- Severity or priority
- Test environment
- Screenshot or supporting evidence when relevant

## 6. Agile Board workflow

The Agile Board is used to organize and follow up the work.

The workflow is based on the following columns:

| Column | Meaning |
|---|---|
| Not Started | Work that has been identified but not started |
| Started | Work that is currently being worked on |
| Done | Work that has been completed |

## 7. Working method

The project group works in the following way:

1. Identify test areas through area tour
2. Create test suites in Reqtest
3. Create test cases for each test suite
4. Prioritize test cases
5. Execute test cases
6. Report bugs when defects are found
7. Select important test cases for the regression test suite
8. Summarize results for the final presentation

## 8. Portfolio reflection

Using Reqtest in this project gives practical experience of working with a test management tool in a structured test project.

The most important learning points are:

- How to divide a system into testable areas
- How to write test cases that others can execute
- How to make bug reports reproducible
- How to track progress visually
- How to connect test planning, execution and reporting
