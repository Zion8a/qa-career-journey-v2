# Agile Reflection – Blekingetrafiken Travel Planner

## 1. Purpose

This document contains a short reflection on agile testing during the practical test project for Blekingetrafiken's web-based travel planner.

The purpose is to connect the project work to agile testing principles, teamwork, communication and continuous improvement.

## 2. Test object

Blekingetrafiken travel planner:

```text
https://blekingetrafiken.se/reseinformation/sok-resa/
```

## 3. Agile context

The project group works with a practical and iterative approach. Instead of trying to define everything perfectly from the start, the group explores the system, creates test ideas, writes test cases, executes tests and updates the work as new information is discovered.

This way of working connects well to agile testing because the focus is on learning, feedback, collaboration and delivering useful information about quality.

## 4. Agile Board workflow

The group uses an Agile Board in Reqtest to organize and follow up the work.

The workflow is based on simple status columns:

| Column | Meaning |
|---|---|
| Not Started | Work that has been identified but not started |
| Started | Work that is currently being worked on |
| Done | Work that has been completed |

This gives the group a visual overview of progress and helps identify what needs attention.

## 5. Testing in an agile team

In an agile team, testing is not only something that happens at the end. Testing supports the whole development process by asking questions, identifying risks and giving feedback early.

In this project, the tester contributes by:

- Exploring the system from a user perspective
- Identifying risks and unclear behavior
- Creating test cases based on important user flows
- Reporting bugs clearly and reproducibly
- Communicating test results to the group
- Helping the team understand quality risks

## 6. Collaboration

The project group divides the work into functional test areas. Each group member or pair can focus on one part of the travel planner, such as search fields, result page, language selection or buttons and links.

This makes the work easier to manage, but it also creates a need for communication. The group must align on how test cases are written, how bugs are reported and what should be included in the regression test suite.

## 7. Feedback and adaptation

An important part of agile testing is to adapt when new information appears.

During the project, the group may discover that:

- Some areas are more complex than expected
- Some functions need more test cases
- Some test cases are unclear and need to be rewritten
- Some bugs affect more than one area
- Some tests should be added to the regression suite

This means the test work should be updated continuously instead of being treated as fixed from the beginning.

## 8. Quality focus

The main quality question in this project is whether users can successfully plan a journey using the travel planner.

From an agile perspective, the most important contribution from testing is not only to find bugs, but also to provide useful information about product quality.

Examples of useful quality information include:

- Which user flows work well
- Which flows are confusing or unstable
- Which bugs are most serious
- Which areas should be prioritized next
- Which tests should be repeated in regression testing

## 9. Personal reflection

This project shows that testing in an agile context requires both structure and flexibility.

The structure comes from test plans, test cases, bug reports and regression suites. The flexibility comes from exploratory testing, discussions in the group and adapting the work based on what is discovered.

A good tester in an agile team needs to combine analytical thinking with communication. It is not enough to find problems. The tester also needs to explain the problem, describe the risk and help the team make better decisions.

## 10. Key learning points

The most important learning points from the agile part of this project are:

- Testing should support the team throughout the project
- Visual planning helps the group understand progress
- Clear communication is essential when several people test the same system
- Test cases and bug reports need to be understandable for others
- Regression testing helps protect important functionality over time
- Quality is a team responsibility, not only the tester's responsibility
