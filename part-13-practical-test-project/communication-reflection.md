# Communication Reflection – Blekingetrafiken Travel Planner

## 1. Purpose

This document reflects on the role of communication during the practical test project for Blekingetrafiken's web-based travel planner.

The purpose is to show how test results, risks, bugs and quality observations can be communicated clearly to support better decisions in a project team.

## 2. Why communication matters in testing

Testing is not only about finding defects. A tester also needs to communicate information about quality, risk and uncertainty in a way that other people can understand and act on.

In this project, communication is important because the group needs to:

- Share observations from the area tour
- Agree on test scope and priorities
- Write test cases that others can execute
- Report bugs clearly and reproducibly
- Summarize test results for the final presentation
- Explain what the findings mean for the quality of the travel planner

## 3. Communication in the project group

The project group works with shared documentation and Reqtest to make the work visible.

Important communication points in the group include:

| Communication area | Purpose |
|---|---|
| Test planning | Agree on scope, risks and priorities |
| Area tour | Share observations and identify test areas |
| Test cases | Make testing repeatable and understandable |
| Bug reports | Describe issues clearly enough to reproduce |
| Agile Board | Show progress and divide work |
| Test summary | Communicate results and quality conclusions |

## 4. Communicating bugs

A bug report should be written so that another person can understand the problem without needing extra explanation.

A good bug report should answer:

- What happened?
- Where did it happen?
- How can it be reproduced?
- What was expected?
- What actually happened?
- How serious is the issue?
- What environment was used?

This helps developers, stakeholders and other testers evaluate the issue and decide what should happen next.

## 5. Communicating test results

Test results should not only describe pass or fail. They should also explain what the result means.

A useful test summary should include:

- What was tested
- What was not tested
- How many test cases were executed
- Which bugs were found
- Which risks remain
- Which areas seem stable
- Which areas need more testing
- Recommended next steps

## 6. Communication challenges

Some communication challenges in a test project can be:

| Challenge | Possible effect | How to handle it |
|---|---|---|
| Different interpretations of requirements | Test cases may become inconsistent | Discuss assumptions and document them |
| Too much detail in bug reports | Important information may be hard to find | Keep reports structured and focused |
| Too little detail in bug reports | Bugs may be hard to reproduce | Include steps, expected result and actual result |
| Unclear priorities | Time may be spent on less important areas | Use risk-based prioritization |
| Many findings close to deadline | Results may be hard to summarize | Update documentation continuously |

## 7. My learning points

This project shows that communication is a central part of the tester role.

The most important learning points are:

- Test documentation must be clear enough for others to use
- Bug reports need to be reproducible, not just descriptive
- Test results should support decisions, not only list findings
- A tester needs to explain both defects and risks
- Good communication helps the team focus on the most important quality issues

## 8. Reflection

In a practical test project, communication connects the technical work with the project goal.

Finding a bug is only useful if the issue is communicated clearly enough for the team to understand its impact and take action. In the same way, a test summary is only useful if it helps the team understand the current quality level and what should be done next.

For me, this project reinforces that a good tester needs both analytical skills and communication skills. The value of testing is created when observations are turned into clear information that improves the product and supports better decisions.
