# Test Plan – Practical Test Project

## 1. Purpose

The purpose of this test plan is to define how the testing will be planned, structured, executed and reported during the practical test project.

The test plan is intended to support the project team in working in a structured way and to make the testing transparent, repeatable and communicable.

## 2. Test object

The test object is the website or application selected by the project group.

Testing will focus on selected areas of the system based on risk, importance and available time.

## 3. Test objectives

The objectives of the testing are to:

- Identify defects that may affect the user experience
- Verify that important functions behave as expected
- Explore the system from a user perspective
- Document test cases and test results clearly
- Report defects in a professional and reproducible way
- Select a smaller regression test suite for future retesting

## 4. Scope

The testing will include:

- Exploratory testing
- Area tour
- Functional test cases
- Negative testing
- Basic usability observations
- Bug reporting
- Regression test selection

## 5. Out of scope

The following areas are not included in the first version of the test project:

- Performance testing
- Security testing
- Automated testing
- Accessibility testing in depth
- Testing on all browsers and devices

These areas may be considered as future improvements.

## 6. Test approach

The testing will be performed using a combination of:

- Exploratory testing
- Structured test cases
- Risk-based prioritization
- Test design techniques
- Group discussion and review

The project starts with an area tour to understand the system and identify important functions, risks and possible problem areas.

## 7. Risks

Potential risks include:

| Risk | Impact | Mitigation |
|---|---|---|
| Unclear requirements | Difficult to know expected result | Use assumptions and document them clearly |
| Limited time | Not all areas can be tested | Prioritize high-value and high-risk areas |
| Different interpretations in the group | Inconsistent testing | Discuss scope and expected results together |
| Test environment changes | Test results may vary | Document date, browser and conditions |

## 8. Test environment

Testing will be documented with relevant environment information, such as:

- Browser
- Device
- Operating system
- Date of test execution
- Tester

## 9. Test deliverables

The project will produce the following test documentation:

- Area tour
- Test cases
- Bug reports
- Regression test suite
- Test summary report

## 10. Exit criteria

The testing can be considered complete when:

- The selected areas have been explored
- Planned test cases have been executed
- Identified bugs have been documented
- A regression test suite has been selected
- Test results have been summarized
- The group can present findings and quality assessment
